# Статус фич: наш плагин vs WorldGuard 7.0.x

## 1. Чат / actionbar / title для deny message (`message-display`)

**В 7.0.x уже есть:**
- `config.yml`: `regions.message-display: chat` (варианты: chat, actionbar, title)
- `ProtectionMessageDisplay.java` — enum CHAT, ACTION_BAR, TITLE
- `BukkitPlayer.sendProtectionMessage()` — выбирает канал по конфигу и поддерживает `&` через `ChatColor.translateAlternateColorCodes('&', message)`

**В нашем плагине (WorldGuardPlus):** то же самое в config и в коде.

Перенос не нужен — в 7.0.x всё реализовано.

---

## 2. Флаг Visualize (команда + кулдаун + право)

**В 7.0.x уже есть:**
- Флаг `Flags.VISUALIZE` (BooleanFlag)
- `VisualizeFlag.java` (session handler):
  - проверяет право `worldguard.visualize`;
  - читает команду из конфига `regions.Visualize` (шаблон с `%region%`);
  - кулдаун из `regions.VisualizeCooldown` (по умолчанию 80 тиков);
  - выполняет команду от имени игрока через `performCommand()` с задержкой (cooldown по времени).
- В конфиге: `Visualize: "/svis wg %region%"`, `VisualizeCooldown: 80`

**В нашем плагине:** в конфиге есть только `Glowing`; класса `VisualizeFlag` нет — при переходе на 7.0.x эта логика уже будет из коробки.

Итог: в 7.0.x флаг работает как нужно (право + кулдаун + команда из конфига). При необходимости можно перенести `VisualizeFlag` и конфиг в наш плагин.

## 3. Цвета в локализации (&, §, HEX {#RRGGBB})

**Сейчас в 7.0.x:**
- В deny-сообщениях уже используется `ChatColor.translateAlternateColorCodes('&', message)`.
- `CommandUtils.replaceColorMacros()` переводит только обратные кавычки `` ` `` в `&` и обрабатывает `&` через LegacyComponentSerializer; **HEX `{#RRGGBB}` не поддерживается**.

Нужно добавить в 7.0.x:
- Единую обработку цветов для строк из языковых файлов: поддержка `&` и при необходимости `§`, плюс **HEX в формате `{#RRGGBB}`** (конвертация в `§x§R§R§G§G§B§B` для 1.16+).
- Применять эту обработку при выдаче строк из локализации (например, в `Localization.get()` / `format()` или в месте отправки в чат/actionbar/title).

---

## 4. Отдельные переводы для флагов в /rg info

Нужно добавить в 7.0.x:
- В языковые файлы секцию с переводами имён флагов (например `flags.names.build`, `flags.names.passthrough` и т.д.).
- В коде вывода информации о регионе (например, `RegionPrintoutBuilder` и связанные места) использовать при отображении флагов не только `flag.getName()`, но и перевод по ключу из этой секции (если есть), чтобы в меню региона отображались локализованные названия флагов.

## 5. Меню «Информация о регионе»: кнопки и 4 варианта

**Реализовано в 7.0.x:**
- **OUTSIDER** = игрок, который **не является ни владельцем (owner), ни участником (member)** региона и не имеет права `worldguard.admin`. То есть любой, кто «не в регионе» с точки зрения членства.
- Четыре варианта меню (ADMIN / OWNER / MEMBER / OUTSIDER), отдельные кнопки [+Владелец]/[+Участник]/[−Владелец]/[−Участник].
- **Конфиг команд** в `config.yml`: секция `region-info.commands` с `add-owner`, `add-member`, `remove-owner`, `remove-member` (поле `command:` с плейсхолдерами `{world}`, `{region_name}`, `{player}`). Если не задано — используются стандартные команды.

Ранее требовалось:

- **Новые кнопки и ключи локализации:**
  - `add-o-label` / `add-m-label` / `remove-o-label` / `remove-m-label` (и соответствующие hover).
  - Отдельные команды для каждой кнопки (по умолчанию `/rg addowner`, `/rg addmember`, `/rg removeowner`, `/rg removemember` с подстановкой региона и т.д.).

- **Настройка команд в конфиге:**
  - Режим: `suggest` (подставить в чат) или `run` (выполнить от имени игрока).
  - Шаблоны команд с плейсхолдерами: `{player}`, `{region_name}`, `{target_player}`, при необходимости `{distance}` или интеграция с PlaceholderAPI.

- **Четыре варианта меню в зависимости от роли:**
  1. Оператор или право `worldguard.admin` — полное меню (тип, приоритет, родитель, флаги, владельцы/участники, телепорт, границы и т.д.).
  2. Владелец региона (без админки) — без части служебных полей (тип/приоритет/родитель/телепорт в центр и т.п., по вашему списку).
  3. Участник (member) без админки — без управления флагами и владельцами, только участники и ограниченный набор кнопок.
  4. Посторонний (не owner/member и не админ) — минимальное меню (например, только название региона, владельцы/участники, без кнопок управления и без флагов/телепорта и т.д.).

Реализация потребует изменений в `RegionPrintoutBuilder`, конфиге и, при необходимости, в модели прав (кто считается owner/member и как передаётся контекст в построитель меню).
